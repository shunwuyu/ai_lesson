# md-wx-vscode 项目规则

本文档定义 md-wx-vscode 项目的开发规范，旨在确保代码质量、开发效率和团队协作的一致性。AI 助手在开发过程中必须严格遵循以下规则。

---

## 一、AI 助手任务执行规范

为确保开发过程的有序性和可控性，AI 助手必须严格遵循以下任务执行规范。

### 1.1 任务范围控制

- **严格按照任务拆分执行**：必须严格按照 `docs/tasks.md` 中定义的任务范围执行，不得超出指定任务的边界。
- **单一任务原则**：每次只执行一个明确指定的任务（如"任务 1.1"、"任务 1.2"等），完成后等待用户确认再进行下一步。
- **禁止自动扩展**：不得基于技术架构文档或其他文档自行扩展任务范围，如果需要扩展需要通知用户确认。

### 1.2 任务指令格式

用户应使用以下格式明确指定任务：

- **明确任务编号**："请执行任务 X.X：[任务名称]"
- **范围限制**："只完成任务 X.X 中列出的具体任务，不要超出范围"
- **停止指令**："完成后等待我确认再进行下一步"

### 1.3 执行验收标准

- **任务完成确认**：每个任务完成后，必须对照 `docs/tasks.md` 中的验收标准进行自检。
- **范围边界检查**：确保所有创建的文件和代码都在指定任务范围内。
- **等待用户确认**：任务完成后必须明确告知完成情况，等待用户确认后再进行下一个任务。

### 1.4 异常处理

- **任务描述不清晰**：如果任务描述不清晰，应先询问具体范围而不是自行决定。
- **依赖关系处理**：如果当前任务依赖其他未完成的任务，应明确指出依赖关系并等待用户指示。
- **超出范围的代码**：如果发现已创建超出任务范围的代码，应主动询问是否需要清理。

## 二、技术选型规范

### 2.1 语言与框架

| 类别 | 选型 | 规范 |
|------|------|------|
| 开发语言 | TypeScript | 所有源代码必须使用 TypeScript，不允许使用 JavaScript |
| 扩展框架 | VS Code Extension API | 使用官方 @types/vscode 类型定义 |
| 构建工具 | Webpack 5.x+ | 配置文件为 webpack.config.js |
| 测试框架 | Jest | 单元测试框架 |
| 扩展测试 | VS Code Extension Test | 集成测试框架 |

### 2.2 禁止使用的技术

| 技术 | 禁止原因 |
|------|----------|
| JavaScript | 项目统一使用 TypeScript |
| 其他构建工具 | 统一使用 Webpack，避免工具碎片化 |
| 未审批的第三方库 | 需确认后再引入 |

## 三、代码风格规范

### 3.1 基础风格

| 规范项 | 规则 | 示例 |
|--------|------|------|
| 缩进 | 2 空格 | 不使用 Tab |
| 引号 | 单引号 | `const name = 'md-wx';` |
| 分号 | 可选 | 不强制要求，但建议统一 |
| 行尾 | LF | 不使用 CRLF |
| 行宽 | 120 字符 | 超过时换行 |

### 3.2 命名规范

| 类型 | 规则 | 示例 |
|------|------|------|
| 变量/函数 | camelCase | `const fileName = '';`<br>`function parseContent() {}` |
| 类 | PascalCase | `class PreviewManager {}` |
| 接口 | PascalCase | `interface UploadOptions {}` |
| 类型别名 | PascalCase | `type MessageHandler = {};` |
| 常量 | UPPER_SNAKE_CASE | `const DEFAULT_PORT = 3000;` |
| 私有成员 | 下划线前缀 | `private _panel: WebviewPanel;` |
| 枚举 | PascalCase | `enum MessageType {}` |
| 文件名 | camelCase（工具）或 PascalCase（类） | `markdown.ts`, `PreviewManager.ts` |

### 3.3 TypeScript 规范

| 规范 | 规则 |
|------|------|
| 严格模式 | 必须启用 `strict: true` |
| 类型定义 | 所有函数参数和返回值必须声明类型 |
| any 禁止 | 禁止使用 `any` 类型，使用 `unknown` 替代 |
| 接口优先 | 优先使用 `interface` 定义对象结构 |
| 类型导出 | 公共类型必须导出 |
| 可空类型 | 使用 `?` 可选语法，而非 `undefined` 联合类型 |

### 3.4 文件组织规范

| 规范 | 说明 |
|------|------|
| 文件长度 | 单文件不超过 500 行，超过则拆分 |
| 导出顺序 | 类型 → 常量 → 函数 → 类 |
| 导入顺序 | 内部模块 → 外部模块 → Node 内置模块 |
| 命名空间 | 不使用 namespace，使用模块系统 |

### 3.5 注释规范

| 类型 | 规范 |
|------|------|
| 行内注释 | 简短说明，放在代码上方，仅用于复杂逻辑 |
| 函数注释 | JSDoc 格式，包含参数、返回值、简要说明 |
| 类注释 | 简要说明类的职责和使用场景 |
| 复杂逻辑 | 必须添加注释说明原因 |
| 注释语言 | 代码注释使用英文，文档使用中文 |

## 四、项目目录结构规范

### 4.1 目录结构

```
md-wx-vscode/
├── .vscode/              # VS Code 配置
├── docs/                 # 项目文档
├── src/                  # 源代码
│   ├── commands/         # 命令定义
│   ├── preview/          # 预览模块
│   ├── services/         # 服务层
│   ├── webview/          # WebView 相关
│   ├── webview-scripts/  # WebView 运行的脚本
│   ├── utils/            # 工具函数
│   └── types/            # 类型定义
├── lib/                  # 本地依赖
│   └── md-wx/            # md-wx 组件（本地）
├── resources/            # 资源文件
├── tests/                # 测试代码
├── trae/                 # 项目规则
│   └── rules/            # AI 执行规则
└── out/                  # 编译输出（生成）
```

### 4.2 目录职责

| 目录 | 用途 | 注意事项 |
|------|------|----------|
| `src/commands/` | VS Code 命令定义与处理 | 每个命令一个文件 |
| `src/preview/` | 预览窗口核心逻辑 | 管理器、面板、同步逻辑 |
| `src/services/` | 可复用的服务层 | 文件监听、上传服务等 |
| `src/webview/` | WebView 提供器与模板 | HTML 模板和 WebView 提供器 |
| `src/webview-scripts/` | WebView 运行的脚本 | 浏览器环境，可访问 DOM |
| `src/utils/` | 纯函数工具 | 不包含业务逻辑 |
| `src/types/` | TypeScript 类型定义 | 统一管理类型 |
| `lib/md-wx/` | md-wx 组件（开发阶段） | 生产阶段改为 npm 包 |

### 4.3 文件放置规则

- **命令文件**：必须放在 `src/commands/` 目录
- **服务文件**：必须放在 `src/services/` 目录
- **工具函数**：必须放在 `src/utils/` 目录
- **类型定义**：放在对应文件或 `src/types/` 目录
- **WebView 相关**：HTML 在 `src/webview/`，脚本在 `src/webview-scripts/`

## 五、NPM 包管理规范

### 5.1 依赖分类

| 类型 | 配置项 | 说明 |
|------|--------|------|
| 生产依赖 | `dependencies` | 运行时必需的依赖 |
| 开发依赖 | `devDependencies` | 仅开发时使用的依赖 |
| 本地依赖 | `file:./path` | 用于 md-wx 开发阶段 |

### 5.2 依赖管理规则

| 规则 | 说明 |
|------|------|
| 版本锁定 | 所有依赖版本必须锁定，不使用 `^` 或 `~` |
| 本地依赖优先 | md-wx 使用本地引用：`"md-wx": "file:./lib/md-wx"` |
| 审核引入 | 引入新包前需确认是否必要 |
| 移除未使用 | 定期清理未使用的依赖 |

### 5.3 禁止的依赖

| 依赖 | 禁止原因 |
|------|----------|
| 未经审批的第三方库 | 安全与合规考虑 |
| 功能重复的库 | 避免依赖膨胀 |

### 5.4 常用依赖版本

| 依赖 | 用途 | 版本要求 |
|------|------|----------|
| @types/vscode | VS Code API 类型 | 最新稳定版 |
| @vscode/extension-test | 测试框架 | 最新稳定版 |
| webpack | 打包构建 | 5.x+ |
| typescript | TypeScript 编译器 | 5.x+ |

## 六、Git 工作流规范

### 6.1 分支策略

| 分支 | 用途 | 说明 |
|------|------|------|
| main | 主分支 | 稳定版本，可部署 |
| develop | 开发分支 | 开发集成分支 |
| feature/* | 功能分支 | 开发新功能 |
| hotfix/* | 修复分支 | 紧急问题修复 |

### 6.2 提交信息规范

| 类型 | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | `feat: 添加复制功能` |
| fix | 修复 Bug | `fix: 修复文件监听失效问题` |
| docs | 文档变更 | `docs: 更新架构文档` |
| style | 代码格式 | `style: 统一代码缩进` |
| refactor | 重构 | `refactor: 重构预览管理器` |
| test | 测试 | `test: 添加单元测试` |
| chore | 构建/工具 | `chore: 更新 webpack 配置` |

### 6.3 提交信息格式

```
<type>: <subject>

<body>

<footer>
```

- **subject**：简短描述，不超过 50 字符
- **body**：详细描述（可选）
- **footer**：关联 Issue 或 PR（可选）

## 七、测试规范

### 7.1 测试类型

| 类型 | 覆盖范围 | 要求 |
|------|----------|------|
| 单元测试 | 工具函数、服务方法 | 覆盖率 > 80% |
| 集成测试 | 模块间交互 | 核心流程全覆盖 |
| E2E 测试 | 完整用户场景 | 关键路径全覆盖 |

### 7.2 测试文件命名

| 类型 | 命名规则 | 示例 |
|------|----------|------|
| 单元测试 | `*.test.ts` | `markdown.test.ts` |
| 集成测试 | `*.integration.test.ts` | `extension.integration.test.ts` |

### 7.3 测试编写规范

- 使用 `describe` 分组测试用例
- 使用 `it` 描述单个测试
- 测试名称使用中文，描述预期行为
- 测试间相互独立，不共享状态

## 八、安全规范

### 8.1 WebView 安全配置

| 配置项 | 要求 |
|--------|------|
| enableScripts | 允许（必需） |
| enableCommandUris | 禁止 |
| localResourceRoots | 严格限制 |
| retainContextWhenHidden | 允许（保留状态） |

### 8.2 数据安全

| 规范 | 说明 |
|------|------|
| 本地文件访问 | 仅允许访问当前工作区 |
| 外部资源 | 不加载外部网络资源 |
| 用户输入 | 进行基本验证 |

### 8.3 禁止行为

| 行为 | 禁止原因 |
|------|----------|
| 硬编码密钥 | 安全风险 |
| 未验证的用户输入 | 安全风险 |
| 直接访问生产数据库 | 合规要求 |

## 九、开发流程规范

### 9.1 开发前准备

1. 确认任务编号和范围
2. 检查依赖的前置任务是否完成
3. 阅读相关文档（需求、架构、任务拆分）

### 9.2 开发中遵守

1. 严格按照任务范围执行
2. 遵循代码风格规范
3. 添加必要的类型定义
4. 编写必要的注释

### 9.3 开发后检查

1. 对照验收标准自检
2. 确认无超出范围的代码
3. 运行必要的测试
4. 等待用户确认

## 十一、异常处理规范

### 11.1 错误处理

| 场景 | 处理方式 |
|------|----------|
| 文件读取失败 | 提示用户并记录日志 |
| 渲染失败 | 显示错误提示，回退到原始内容 |
| 通信失败 | 重试机制，超时后提示 |
| 未预期错误 | 记录详细信息，提示用户 |

### 11.2 用户提示

- 使用 VS Code 信息提示 API
- 提示信息简洁明确
- 提供解决建议

## 十二、版本管理规范

### 12.1 版本格式

遵循语义化版本：`MAJOR.MINOR.PATCH`

- **MAJOR**：不兼容的 API 变更
- **MINOR**：向下兼容的功能新增
- **PATCH**：向下兼容的问题修复

### 12.2 发布流程

1. 更新版本号（package.json）
2. 更新 CHANGELOG.md
3. 运行测试确保通过
4. 提交并打标签
5. 发布

---

## 十三、规范遵守检查

AI 助手在执行任务前，必须检查以下清单：

- [ ] 任务编号和范围是否明确
- [ ] 前置依赖任务是否完成
- [ ] 是否有超出任务范围的代码
- [ ] 代码风格是否符合规范
- [ ] 类型定义是否完整
- [ ] 是否有必要的错误处理
- [ ] 是否遵循安全规范
