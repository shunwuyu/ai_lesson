# VS Code Markdown 微信公众号预览插件 - 技术架构设计文档

## 1. 概述

本文档描述 md-wx-vscode 项目的整体技术架构，包括目录结构规范、编码规范、核心模块设计以及 md-wx 组件在 VS Code 插件环境下的适配方案。

## 2. 技术选型

### 2.1 核心技术栈

| 技术 | 用途 | 选型理由 |
|------|------|----------|
| TypeScript | 主要开发语言 | 类型安全，VS Code 扩展开发标准 |
| VS Code Extension API | 插件开发框架 | VS Code 官方扩展能力 |
| WebView API | 预览窗口渲染 | 提供独立渲染环境，支持浏览器运行时 |
| Node.js | 后端逻辑处理 | VS Code 扩展运行环境 |
| md-wx | Markdown 渲染组件 | 微信公众号样式渲染核心能力 |

### 2.2 依赖管理

| 依赖 | 版本要求 | 用途 |
|------|----------|------|
| @types/vscode | 最新稳定版 | VS Code API 类型定义 |
| @vscode/extension-test | 最新稳定版 | 测试框架 |
| webpack | 5.x+ | 打包构建 |
| md-wx | 本地路径引用 | Markdown 渲染（开发阶段） |

### 4.2 模块职责说明

#### 4.2.1 Extension Entry (extension.ts)
- 职责：扩展的激活与停载入口
- 职责范围：
  - 注册 VS Code 命令
  - 初始化服务模块
  - 配置事件监听器
  - 处理扩展停载清理

#### 4.2.2 Command Module
- 职责：VS Code 命令定义与处理
- 职责范围：
  - 注册 "打开预览" 命令
  - 处理命令执行逻辑
  - 验证当前上下文

#### 4.2.3 PreviewManager
- 职责：预览窗口的核心管理器
- 职责范围：
  - 创建与管理 PreviewPanel
  - 协调内容同步
  - 处理预览生命周期

#### 4.2.4 PreviewPanel
- 职责：预览面板的具体实现
- 职责范围：
  - 创建 WebView 面板
  - 设置 WebView 内容
  - 处理 WebView 消息通信

#### 4.2.5 ContentSync
- 职责：编辑器内容与预览的同步
- 职责范围：
  - 监听编辑器内容变化
  - 将变更通知到 WebView
  - 处理防抖与节流

#### 4.2.6 FileWatcher
- 职责：文件变更监听服务
- 职责范围：
  - 监听当前 Markdown 文件变更
  - 触发内容更新通知
  - 处理外部文件修改

#### 4.2.7 WebViewProvider
- 职责：WebView 的提供与配置
- 职责范围：
  - 生成 WebView HTML 模板
  - 注入运行时脚本
  - 配置 WebView 安全策略

#### 4.2.8 preview.js
- 职责：在 WebView 环境中运行的脚本
- 职责范围：
  - 接收来自扩展的消息
  - 调用 md-wx 组件进行渲染
  - 处理复制功能
  - 处理用户交互

#### 4.2.9 ImageUploader（中优先级）
- 职责：图片上传服务
- 职责范围：
  - 检测剪贴板图片
  - 上传到存储服务
  - 返回图片链接

## 5. md-wx 组件适配方案

### 5.1 适配挑战分析

md-wx 组件原为浏览器端组件，在 VS Code 插件环境中运行面临以下挑战：

| 挑战 | 说明 | 解决方案 |
|------|------|----------|
| 运行环境差异 | VS Code 扩展运行在 Node.js 环境 | 使用 WebView 提供浏览器运行时 |
| 依赖注入 | 需要在 WebView 中加载 md-wx | 将 md-wx 打包到 WebView 资源中 |
| 通信机制 | 扩展与 WebView 需要通信 | 使用 VS Code WebView 消息 API |
| DOM 依赖 | md-wx 依赖 DOM 操作 | WebView 提供 DOM 环境 |

### 5.2 架构适配方案

#### 5.2.1 本地组件集成策略

由于 md-wx 尚未发布至 NPM，采用以下集成策略：

```
md-wx 组件存放位置: lib/md-wx/

集成方式:
┌─────────────────────────────────────────────────────┐
│  1. 开发阶段: 将 md-wx 源码放入 lib/md-wx/          │
│  2. 构建阶段: 将 md-wx 打包进 webview-scripts/      │
│  3. 运行阶段: 在 WebView 中引入并初始化              │
└─────────────────────────────────────────────────────┘
```

#### 5.2.2 WebView 运行时架构

```
┌─────────────────────────────────────────────────────────────┐
│                     VS Code Extension                       │
│                      (Node.js 环境)                         │
│                                                             │
│  1. 读取当前 Markdown 文件内容                              │
│  2. 通过 WebView API 发送消息到 WebView                      │
│  3. 接收 WebView 的响应（如复制结果）                        │
└────────────────────┬────────────────────────────────────────┘
                     │
                     │ VS Code WebView 消息通信
                     │ (postMessage / onDidReceiveMessage)
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                      WebView Panel                           │
│                   (浏览器运行时环境)                         │
│                                                             │
│  1. 接收来自扩展的 Markdown 内容                            │
│  2. 调用 md-wx.renderHtml() 渲染                            │
│  3. 将渲染结果插入 DOM                                       │
│  4. 处理复制等交互事件                                        │
│  5. 将结果反馈回扩展                                         │
│                                                             │
│  ┌──────────────────────────────────────┐                  │
│  │  md-wx Component (浏览器端运行)      │                  │
│  │  - renderHtml()                      │                  │
│  │  - convertSvgToPng()                 │                  │
│  │  - GitHub 风格渲染                   │                  │
│  │  - KaTeX LaTeX 支持                  │                  │
│  └──────────────────────────────────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

#### 5.2.3 组件加载方案

| 方案 | 描述 | 优点 | 缺点 | 选择 |
|------|------|------|------|------|
| 源码引入 | 将 md-wx 源码直接复制到项目 | 无需构建，开发快速 | 难以维护，版本管理困难 | ❌ |
| npm link | 使用 npm link 链接本地包 | 标准方式，易于调试 | 依赖本地环境 | ❌ |
| 相对路径引用 | package.json 中使用 file: 协议 | 标准 npm 语法，版本可管理 | 需要 npm install | ✅ |
| 打包集成 | 构建时将 md-wx 打包进输出 | 运行时无外部依赖 | 构建复杂度增加 | ✅ |

**最终方案**：采用相对路径引用 + 打包集成

1. **开发阶段**：package.json 中使用 `file:./lib/md-wx` 引用
2. **构建阶段**：通过 webpack 将 md-wx 打包进 webview-scripts
3. **生产阶段**：md-wx 发布到 NPM 后，改为正常 npm 引用

#### 5.2.4 消息通信协议

扩展与 WebView 之间采用消息通信：

```typescript
// 扩展 → WebView 消息类型
type ExtensionToWebViewMessage =
  | { type: 'update'; content: string }        // 更新 Markdown 内容
  | { type: 'requestCopy' };                  // 请求复制内容

// WebView → 扩展消息类型
type WebViewToExtensionMessage =
  | { type: 'copyResult'; success: boolean }  // 复制结果
  | { type: 'error'; message: string };       // 错误信息
```

## 6. 编码规范

### 6.1 代码风格

| 规范类别 | 标准 | 说明 |
|----------|------|------|
| 缩进 | 2 空格 | 统一使用 2 空格缩进 |
| 引号 | 单引号 | 字符串使用单引号 |
| 分号 | 可选 | 不强制要求分号 |
| 命名 | camelCase | 变量和函数使用驼峰命名 |
| 类名 | PascalCase | 类名使用帕斯卡命名 |
| 常量 | UPPER_SNAKE_CASE | 常量使用大写下划线 |
| 注释 | 英文 | 代码注释使用英文 |

### 6.2 TypeScript 规范

| 规范 | 要求 |
|------|------|
| 严格模式 | 启用 strict: true |
| 类型定义 | 所有函数参数和返回值必须声明类型 |
| any 禁止 | 禁止使用 any 类型，使用 unknown 替代 |
| 接口优先 | 优先使用 interface 定义对象结构 |
| 类型导出 | 公共类型必须导出 |

### 6.3 文件组织规范

| 规范 | 要求 |
|------|------|
| 文件命名 | 使用 PascalCase（类文件）或 camelCase（工具文件） |
| 导出顺序 | 类型 → 常量 → 函数 → 类 |
| 导入顺序 | 内部模块 → 外部模块 → Node 内置模块 |
| 文件长度 | 单文件不超过 500 行 |

### 6.4 注释规范

| 类型 | 规范 |
|------|------|
| 行内注释 | 简短说明，放在代码上方 |
| 函数注释 | JSDoc 格式，说明参数、返回值、用途 |
| 类注释 | 简要说明类的职责 |
| 复杂逻辑 | 必须添加注释说明原因 |
