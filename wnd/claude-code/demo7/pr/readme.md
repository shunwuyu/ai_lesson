## 什么是pr

Pull Request (PR) 是现代软件开发中协作的核心流程。简单来说，就是“你修改代码 -> 请求合并 -> 他人审查 -> 正式入库”的过程。

为了让你更直观地理解，我将整个流程拆解为 5 个关键步骤，并配合一个具体的“修复网站错别字”的例子来说明。

Pull Request 标准操作流程
假设场景：你想为一个开源项目（比如一个名为 MyWebsite 的项目）修复首页上的一个错别字。
第一步：准备环境 (Fork & Clone)
你通常没有权限直接修改别人的主仓库，所以需要先“复制”一份到自己的账户下。
Fork (复刻)： 在 GitHub 网页上点击原项目页面的 "Fork" 按钮。这会在你的账户下创建一个完全一样的副本。
Clone (克隆)： 将你账户下的这个副本下载到你的本地电脑。
命令示例： git clone https://github.com/你的用户名/MyWebsite.git
第二步：分支开发 (Branch & Edit)
永远不要直接在主分支（main/master）上修改代码，这是大忌。
新建分支： 创建一个专门用于修复这个错别字的新分支。
命令示例： git checkout -b fix/typo-homepage
修改代码： 在本地编辑器中打开文件，把错别字改对。
第三步：提交与推送 (Commit & Push)
将你的修改保存并上传到你的远程仓库。
提交更改： 告诉 Git 你改了哪里，并写清楚修改原因。
命令示例： git add . 然后 git commit -m "fix: 修复首页欢迎语的拼写错误"
推送代码： 将这个新分支上传到 GitHub。
命令示例： git push origin fix/typo-homepage
第四步：发起请求 (Create PR)
现在代码在你手里，你需要告诉原作者：“我改好了，你要不要看看？”
网页操作： 回到 GitHub 网页（通常是你 Fork 的那个仓库页面）。你会看到一个黄色的提示条 "Compare & pull request"，点击它。
填写信息：
标题： 简明扼要，如“修复首页拼写错误”。
描述： 详细说明你改了什么，为什么要改，甚至可以贴上修改前后的对比图。
发送： 点击 "Create pull request" 按钮。
第五步：审查与合并 (Review & Merge)
这是协作发生的地方。
代码审查： 项目维护者会收到通知，查看你的代码。他们可能会评论：“这里改得不错，但是那边格式有点问题”。
迭代修改（如果需要）： 如果维护者提出了意见，你只需要在本地同一个分支上继续修改，然后再次 commit 和 push。PR 页面会自动更新你的新代码，不需要重新发起 PR。
合并： 维护者觉得没问题了，点击 "Merge" 按钮。你的代码就正式成为了原项目的一部分！🎉

流程关键点总结表
表格
阶段	核心动作	目的
准备	Fork & Clone	获得代码副本，建立开发环境
开发	Branch (新建分支)	隔离修改，保证主分支稳定
提交	Commit & Push	保存快照并上传至云端
协作	Pull Request	发起合并请求，触发代码审查
结束	Merge	代码正式入库，任务完成

git branch -d fix/typo-homepage
