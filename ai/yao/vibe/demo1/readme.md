# 轻松上手的Vibe Coding 指南

github 上有一个[Vibe编程指南项目](https://github.com/tradecatlabs/vibe-coding-cn)，给出了清晰的Vibe 路径。

- 极客先star

## 现实情况

很多人想用Ai 去写代码，但Ai 生成的要不是“幻觉代码”， 看起来像那么回事，一跑就报错。
要么就是屎山代码， 能跑，但乱的你想改都不知道从哪下手。

这个指南不教怎么写代码，而叫你去智慧AI去写出靠谱的代码。

## 操作核心方法

- 第一步：规划就是一切

不要上来就让Ai 去写代码， 先强制它去读懂你的技术栈和项目规划。

就好像你不会让新来的员工直接干活，得先让他熟悉公司业务流程一样， AI 生成代码逻辑清晰多了。

### demo

场景：用 React + Tailwind 实现一个任务清单（待办）页面

普通人错误提问（直接上手写代码，极易幻觉、屎山）

帮我写一个 React 待办清单页面，支持新增、删除任务。

先规划，再编码

```
遵守胶水编程思维：优先复用成熟方案，避免凭空造逻辑。
第一阶段：只做规划，禁止输出任何代码。
1. 确认技术栈：React 18 + TailwindCSS + useState
2. 梳理功能边界：
   - 新增待办、删除待办、切换完成状态
   - 不做本地持久化、筛选、拖拽，缩减范围
3. 拆分模块（乐高组件）：
   ① 输入框组件 ② 待办条目组件 ③ 列表容器
4. 定义数据流：
   useState存储tasks数组，数据结构：{id, text, completed}
5. 列出潜在风险：不要自创唯一ID方案，优先使用Date.now简易实现
6. 输出这份完整规划，等待我确认无误后，再分段实现代码。
```

背后逻辑（贴合 Vibe 编码指南）
先划定边界：防止 AI 擅自多加功能，无限膨胀成屎山；
强制模块拆分：引导 AI 复用独立组件，而不是堆在单个文件；
预先约定数据结构：从根源减少 AI 参数、字段幻觉；
先评审规划，再写代码：规划不合格直接调整，不用写完代码再大规模重构。



第二步： 胶水编程思维

能抄不写， 能连不造。 指挥Ai 去找那些已经被验证过千万次的成熟组件， 拼乐高一样，把它给组装起来。

这样出来的代码， 稳定性搞、幻觉大大降低。

胶水本身不创造零件，只负责把现成零件粘在一起， 你只负责写衔接、调用、流转的 “粘合代码”，把各个模块连通。

轮子别人造好，你只做胶水；胶水不生产零件，只连通零件。


胶水编程实战案例（承接上一个 React 待办项目，连贯演示）
场景：我们已经完成项目规划，现在正式编码，对比两种方式
❌ 传统写法（从零造零件，高幻觉风险）
直接提示词：
帮我写 React 待办清单的拖拽排序功能
AI 很可能凭空手写一套拖拽逻辑，自行实现坐标监听、排序算法。
问题：手写拖拽逻辑边界 case 极多，容易出现幻觉 bug，代码难以维护。

```
遵循胶水编程原则：绝不从零自研底层逻辑，优先选用社区长期验证成熟开源组件。
当前需求：给待办列表增加拖拽排序
1. 先调研：React生态成熟拖拽库，优先选用 react-beautiful-dnd（生产广泛使用）
2. 不要自己手写拖拽底层代码，只做粘合工作
3. 输出内容顺序：
   ① 安装依赖命令
   ② 把现有TodoList组件 和 react-beautiful-dnd 进行衔接改造
   ③ 只编写模块之间适配、数据流转的粘合代码
4. 禁止改写我已经写好的任务新增、删除原有逻辑，只新增粘合代码
```


第三步： 用元方法论让Ai 自我进化

简单来说就是用一个α（Alpha） 提示词， 去生成提示词， 再用一个Ω欧米伽提示词去优化
这个提示词。通过这种左脚踩右脚，这样的一个递归循环，让你的Ai 系统不断地自我进化。

持续的优化和改进。

α 提示词：全局底层约束。 新开对话第一条，先发这段α 提示词
```
【α顶层永久规则，全程生效，所有任务优先遵守】
1. 严格执行胶水编程思维：能连不造，优先复用久经验证的成熟开源组件，禁止凭空从零实现底层逻辑。
2. 任何开发需求：先输出整体规划，确认规划无误后，再分段编写代码，禁止直接一次性输出大段代码。
3. 警惕代码幻觉：优先给出可落地、社区通用方案，不编造不存在的API、函数。
4. 代码模块化，杜绝屎山，只编写衔接适配的粘合代码。
后续所有需求，都基于以上规则执行。
```

α 规定「怎么干活」，Ω 规定「交付结果必须达到什么标准」。

【Ω 最终交付验收标准，输出成果前强制自检】
1. 所有代码优先复用成熟开源组件，不存在手写冗余底层逻辑，符合胶水编程思想。
2. 杜绝幻觉：所有API、组件、函数真实存在，无编造接口与参数。
3. 代码模块化拆分，粘合代码与业务逻辑边界清晰，不产生单体屎山。
4. 附带简要使用说明、依赖安装命令、已知风险提示。
5. 输出前自查：若违反以上任意一条，主动优化后再给出最终代码。
